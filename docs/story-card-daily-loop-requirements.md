# 스토리 카드 기반 일일 커플 상호작용 요구사항

작성일: 2026-06-29

이 문서는 기존 `오늘의 질문 -> 각자 답변` 중심 구조를 `스토리 카드 작성 -> 질문 생성 -> 각자 답변` 구조로 전환하기 위한 요구사항을 정리한다.

## 1. 목적

- 질문 이전에 감정과 상황을 시각적으로 교환할 수 있어야 한다.
- 사진, 그림, 짧은 글을 통해 더 자연스럽고 직관적인 애정 표현이 가능해야 한다.
- 질문이 현재 커플의 실제 맥락을 더 잘 반영하도록 시작점을 바꿔야 한다.
- 홈 화면과 위젯이 단순 알림 수단이 아니라 관계 상호작용의 진입점이 되어야 한다.

## 2. 핵심 개념

### 2.1 스토리 카드

- 스토리 카드는 하루 단위로 각 사용자가 작성하는 시각 콘텐츠이다.
- 스토리 카드는 하나의 세로형 폴라로이드 프레임 결과물이다.
- 사용자는 하나의 캔버스에서 아래 요소를 자유롭게 조합할 수 있어야 한다.
  - 사진 추가
  - 자유 드로잉
  - 짧은 텍스트 추가
- 사진, 그림, 글 중 하나 이상이 포함되면 유효한 카드로 간주한다.
- 아래 형태를 모두 허용한다.
  - 사진만 있는 카드
  - 그림만 있는 카드
  - 글만 있는 카드
  - 사진 + 그림
  - 사진 + 글
  - 그림 + 글
  - 사진 + 그림 + 글

### 2.2 하루 단위 루프

하루의 메인 루프는 아래 순서로 동작한다.

1. 사용자 A가 스토리 카드를 작성한다.
2. 사용자 B가 스토리 카드를 작성한다.
3. 두 사람 모두 완료되면 질문이 생성된다.
4. 각 사용자가 생성된 질문에 텍스트 답변을 작성한다.

## 3. 범위

### 3.1 이번 범위에 포함

- 스토리 카드 작성 기능
- 스토리 카드 조회 및 공개 규칙
- 질문 생성 시점 변경
- 텍스트 답변 플로우 유지
- 홈 화면 구조 변경
- 홈 화면 내 캐릭터 / 녹음 / 보관함 배치 재정의
- 캘린더 기록 구조 변경
- 위젯 미리보기 및 앱 진입 기능
- 관련 알림 기능

### 3.2 이번 범위에 포함하지 않음

- 위젯 내 직접 편집
- 질문 생성 로직의 AI 고도화
- 답변에 이미지 또는 그림 첨부
- 기존 placeholder 질문 생성 구조의 고도화
- 인앱 결제와 연결된 추가 기능

## 4. 기능 요구사항

### 4.1 하루 콘텐츠 구조

- 하루에 하나의 커플 공용 진행 단위가 존재해야 한다.
- 각 사용자는 하루에 자기 스토리 카드 1개를 작성할 수 있어야 한다.
- 두 사람의 스토리 카드가 모두 존재할 때만 질문이 생성되어야 한다.
- 질문 생성 전까지는 해당 하루의 상태를 스토리 카드 작성 단계로 간주한다.
- 질문 생성 후에는 질문 답변 단계로 전환되어야 한다.

### 4.2 스토리 카드 작성

- 사용자는 스토리 카드 편집 화면에서 콘텐츠를 작성할 수 있어야 한다.
- 캔버스와 최종 미리보기는 세로형 4:5 폴라로이드 프레임이어야 한다.
- 폴라로이드 프레임은 상단 1:1 사진 영역과 하단 고정 문구 영역으로 구성해야 한다.
- 사용자는 편집 화면에서 아래 작업을 수행할 수 있어야 한다.
  - 사진 추가
  - 자유 드로잉
  - 짧은 텍스트 추가
- 카드에는 위 세 요소 중 하나 이상이 반드시 포함되어야 한다.
- 빈 카드 저장은 허용되지 않아야 한다.
- 새 카드 작성은 앱 내부 카메라 화면에서 시작해야 한다.
- 카메라 화면 왼쪽 하단에는 다른 도구와 분리된 갤러리 선택 버튼이 있어야 한다.
- 카메라 화면에서 텍스트 또는 그리기를 선택하면 촬영을 건너뛰고 흰색 또는 검은색 빈 캔버스로 꾸미기 단계에 진입해야 한다.
- 촬영하거나 갤러리에서 선택한 사진은 폴라로이드의 1:1 사진 영역 중앙을 기준으로 채워져야 한다.
- 사용자는 사진을 확대, 축소, 이동할 수 있어야 하며 이동 결과로 캔버스 일부가 비어도 된다.
- 사진 이동과 확대·축소는 두 손가락 제스처로만 수행해야 한다.
- 사진은 1:1 사진 영역 밖에 그려지지 않아야 한다.
- 자유 텍스트와 드로잉은 사진 영역에 제한하지 않고 폴라로이드 프레임 전체에 배치할 수 있어야 한다.
- 사용자는 폴라로이드 하단에 고정되는 선택 문구를 추가할 수 있어야 한다.
- 고정 문구는 최대 50자, 최대 2줄이며 색상, 정렬, 글꼴을 사용자가 변경할 수 없어야 한다.
- 고정 문구는 자유 텍스트 레이어와 분리해 저장하고 이동, 확대·축소, 회전을 지원하지 않아야 한다.
- 사진, 자유 텍스트, 드로잉이 없는 고정 문구만으로는 빈 카드 저장 조건을 충족하지 않은 것으로 간주한다.
- 꾸미기 단계에서는 사진 교체를 지원하지 않는다. 다른 사진을 쓰려면 현재 편집 내용을 폐기하고 카드 작성을 다시 시작해야 한다.
- 텍스트 도구를 선택하면 별도 모달 없이 현재 카드 위에 반투명한 검정 입력 오버레이를 표시하고 키보드를 즉시 열어야 한다.
- 입력 오버레이는 빈 커서에서 시작하며 텍스트 기본 색상은 흰색이어야 한다.
- 입력 오버레이는 키보드 위에 텍스트 색상 팔레트를 제공해야 하며, 선택한 색상은 입력 중인 텍스트와 생성되는 레이어에 즉시 반영되어야 한다.
- 입력을 완료하면 텍스트 레이어를 캔버스 중앙에 생성해야 한다. 사용자는 생성 후 제스처로 위치와 크기를 조정한다.
- 입력을 취소하거나 빈 문자열로 완료하면 텍스트 레이어를 생성하지 않아야 한다.
- 기존 텍스트를 탭해 수정하거나 삭제하는 기능은 제공하지 않아야 한다. 수정이 필요하면 기존 레이어를 삭제하고 새 레이어를 생성한다.
- 텍스트 레이어는 한 손가락 이동과 두 손가락 확대·축소·자유 회전을 지원해야 한다.
- 두 손가락 중 하나만 텍스트 위에서 시작해도 해당 텍스트의 확대·축소·회전을 우선해야 한다.
- 두 손가락으로 텍스트를 변환하던 중 텍스트 위에서 시작한 손가락을 먼저 떼더라도, 남은 손가락을 놓을 때까지 해당 텍스트의 이동을 계속할 수 있어야 한다.
- 텍스트를 한 손가락으로 드래그하기 시작하면 화면 하단 중앙에 휴지통 아이콘을 표시해야 한다.
- 텍스트가 휴지통 삭제 영역에 들어오면 아이콘 색상과 상태를 변경해야 한다. 해당 상태에서 마지막 손가락을 놓으면 텍스트 레이어를 삭제하고, 영역 밖에서 놓으면 이동한 위치에 유지해야 한다.
- 텍스트 최대 배율은 기본 크기의 8배까지 허용해야 한다.
- 텍스트 레이어는 최대 10개, 레이어당 최대 500자, 전체 최대 5,000자로 제한한다.
- 그리기 도구는 자유 드로잉과 색상 팔레트, 굵기 선택을 지원해야 하며 최대 굵기는 캔버스 짧은 변의 8%까지 허용해야 한다.
- 그리기 도구는 드로잉 레이어만 부분 삭제하는 지우개를 지원해야 한다.
- 그리기 도구는 마지막으로 완료한 펜 또는 지우개 스트로크를 되돌릴 수 있어야 한다.
- 그리기 도구에는 명시적인 `완료` 동작이 있어야 한다. 사진 카드에서는 완료 후 배경 조정 상태로, 사진 없는 카드에서는 중립 상태로 전환해야 한다.

### 4.3 스토리 카드 공개 규칙

- 한쪽 사용자만 카드를 올린 상태에서도 상대방은 그 카드를 즉시 볼 수 있어야 한다.
- 카드 공개는 질문 생성 이후가 아니라 업로드 직후 이루어져야 한다.
- 상대방은 상대 카드를 본 뒤 본인의 카드 작성으로 자연스럽게 이어질 수 있어야 한다.

### 4.4 수정 및 삭제 규칙

- 질문 생성 전에는 사용자가 본인 스토리 카드를 수정할 수 있어야 한다.
- 질문 생성 전에는 사용자가 본인 스토리 카드를 삭제할 수 있어야 한다.
- 질문 생성 전에는 삭제 후 재등록할 수 있어야 한다.
- 질문 생성 후에는 두 사용자의 해당 날짜 스토리 카드가 수정 및 삭제 불가 상태가 되어야 한다.

### 4.5 질문 생성

- 질문은 두 사람의 스토리 카드가 모두 업로드된 직후 생성되어야 한다.
- 이번 범위에서는 질문 생성 결과를 기존 placeholder 구조로 유지한다.
- 즉, 질문 생성 품질이나 생성 방식은 지금 단계에서 고도화하지 않는다.
- 이번 변경의 핵심은 질문 생성 로직 교체가 아니라 질문 생성 시점의 이동이다.

### 4.6 질문 답변

- 질문 생성 이후 답변 방식은 기존과 동일하게 텍스트 입력만 지원해야 한다.
- 사용자는 생성된 질문에 대해 텍스트 답변을 작성할 수 있어야 한다.
- 기존 답변 공개 규칙은 유지한다.
- 두 사람 모두 답변을 완료해야 서로의 답변이 공개되어야 한다.

## 5. 화면별 요구사항

### 5.1 홈 화면

- 홈 화면의 중심 콘텐츠는 더 이상 오늘의 질문이 아니어야 한다.
- 홈 화면은 여전히 사용자가 직접 그린 캐릭터가 머무는 방 구조를 유지해야 한다.
- 캐릭터는 홈에서 사라지면 안 되며, 여전히 메인 시각 요소로 남아야 한다.
- 홈 화면은 오늘의 스토리 카드 상태를 중심으로 보여줘야 한다.
- 스토리 카드는 홈 전체를 대체하는 전면 화면이 아니라, 방 안의 중심 오브젝트로 배치되어야 한다.
- 홈에서 캐릭터 오브젝트와 스토리 카드 오브젝트는 서로 다른 오브젝트로 유지되어야 한다.
- 아무도 아직 오늘 카드를 작성하지 않은 경우, 홈은 오늘 카드가 비어 있는 상태와 작성 진입점을 보여줘야 한다.
- 내가 먼저 오늘 카드를 작성한 경우, 홈은 내 카드와 상대 대기 상태를 함께 보여줘야 한다.
- 한쪽만 오늘 카드를 작성한 경우, 상대가 작성한 오늘 카드가 바로 보여야 한다.
- 두 사람 모두 오늘 카드를 작성했지만 질문 생성이 아직 완료되지 않은 짧은 구간에는 `질문 생성 중` 상태를 보여줘야 한다.
- 두 사람 모두 오늘 카드를 작성한 경우, 질문 생성 상태를 반영해야 한다.
- 질문이 생성된 이후에는 해당 질문으로 진입할 수 있어야 한다.
- 캐릭터 편집 진입은 홈에서 계속 바로 접근 가능해야 한다.
- 홈의 녹음 기능도 방 안에 함께 배치되어야 하며, 별도 카드형 패널처럼 보이면 안 된다.
- 홈의 녹음 인터랙션은 하나의 메인 버튼으로 처리되어야 한다.
  - 짧게 터치하면 현재 녹음을 재생하거나 일시정지해야 한다.
  - 길게 누르면 새 녹음을 시작해야 한다.
  - 녹음 중 위로 드래그하면 취소할 수 있어야 한다.
- 홈에는 녹음 보관함으로 이동하는 별도 버튼이 있어야 한다.
- 홈에서는 녹음 시각, 길이, 발신자 같은 상세 메타 정보를 노출하지 않아야 한다.
- 현재 재생 중 길게 눌러 새 녹음을 시작하는 경우, 기존 재생은 자동으로 중지되어야 한다.
- 현재 재생 가능한 녹음이 없는 상태에서 메인 녹음 버튼을 짧게 누르면, 재생 가능한 녹음이 없다는 짧은 피드백이 제공되어야 한다.

### 5.2 스토리 카드 편집 화면

- 사용자는 홈 또는 위젯에서 편집 화면으로 진입할 수 있어야 한다.
- 편집 화면은 세로형 4:5 폴라로이드 프레임을 화면에서 가능한 가장 크게 제공해야 한다.
- 편집 화면은 사진, 드로잉, 텍스트를 같은 결과물 위에서 조합할 수 있어야 한다.
- 편집 화면에서는 상하 스크롤이 발생하면 안 된다.
- 편집 도구, 색상 팔레트, 굵기 선택, 지우개, 되돌리기, 그리기 완료는 캔버스 위에 오버레이되어야 한다.
- 텍스트 입력 오버레이는 편집 화면 전체를 덮되 카드가 비쳐 보이는 반투명 검정 배경을 사용해야 한다.
- 텍스트 입력 중에는 화면 상단 완료 동작과 키보드 완료 동작을 제공하고, 시스템 뒤로가기는 현재 입력을 취소한 뒤 편집 화면에 머물러야 한다.
- 텍스트 입력 오버레이와 드래그 중 휴지통은 미리보기 캡처 영역 밖의 편집 UI로 취급해 저장 결과물에 포함하지 않아야 한다.
- 별도 자르기 버튼은 제공하지 않아야 한다. 사진이 있고 그리기 중이 아니라면 빈 영역에서 시작한 두 손가락 제스처로 배경을 조정할 수 있어야 한다.
- 카메라 단계에서 뒤로가기를 누르면 카드 작성 화면을 종료해야 한다.
- 꾸미기 단계에서 변경 사항이 없을 때 뒤로가기를 누르면 확인 창 없이 카메라 단계로 돌아가야 한다.
- 꾸미기 단계에서 변경 사항이 있을 때 뒤로가기를 누르면 `수정 내용을 삭제하시겠어요?` 확인 창을 보여줘야 한다.
- 확인 창은 `삭제`와 `계속 수정`을 제공해야 한다.
- `삭제`를 선택하면 현재 편집 중인 사진, 그림, 텍스트를 모두 제거하고 꾸미기 단계에 남아야 한다. 그 상태에서 다시 뒤로가면 카메라 단계로 돌아가야 한다.
- `계속 수정`을 선택하면 확인 창만 닫고 꾸미기 단계에 남아야 한다.
- 시스템 뒤로가기와 화면의 뒤로가기 버튼은 같은 동작 규칙을 사용해야 한다.
- 질문 생성 후에는 해당 날짜 카드가 잠겨 편집 불가 상태가 되어야 한다.

### 5.3 질문 화면

- 질문 화면은 기존 구조를 최대한 유지한다.
- 다만 질문이 먼저 존재하는 구조가 아니라, 스토리 카드 완료 이후 생성된 질문을 보여주는 구조로 바뀌어야 한다.
- 답변 작성 화면도 기존 텍스트 답변 구조를 유지한다.
- 홈과 캘린더 어디에서 진입하더라도, 오늘 질문이 편집 가능 상태이면 답변 작성 및 수정까지 이어질 수 있어야 한다.
- 질문 답변 작성 또는 수정이 끝난 뒤에는, 사용자가 진입했던 출처로 자연스럽게 복귀해야 한다.

### 5.4 캘린더 화면

- 캘린더 기록 단위는 기존 질문 중심 기록에서 스토리 카드 중심 기록으로 변경되어야 한다.
- 사용자는 날짜별로 아래 정보를 확인할 수 있어야 한다.
  - 그날의 스토리 카드
  - 생성된 질문
  - 두 사람의 답변
- 하루 기록의 대표 단위는 질문이 아니라 스토리 카드여야 한다.
- 월간 셀 탭의 기본 동작은 해당 날짜의 선택 상태와 상세 패널 내용을 현재 화면 안에서 갱신하는 것이어야 한다.
- 오늘 날짜의 질문이 편집 가능 상태라면, 캘린더를 통해 진입한 경우에도 답변 작성 및 수정이 가능해야 한다.

### 5.5 위젯

- 위젯은 배경화면에서 현재 스토리 카드 상태를 미리보기로 보여줘야 한다.
- 위젯의 미리보기 규칙은 홈 화면 표시 규칙과 분리되어야 한다.
- 오늘 스토리 카드가 존재하는 경우, 위젯은 오늘의 현재 스토리 카드 상태를 보여줘야 한다.
- 아무도 아직 오늘 카드를 작성하지 않은 경우, 위젯은 상대가 최근 올린 스토리 카드를 미리보기로 보여줄 수 있어야 한다.
- 상대가 지금까지 한 번도 스토리 카드를 올린 적이 없는 경우, 위젯은 상대 카드가 없는 대기 상태를 보여줘야 한다.
- 이 경우 위젯은 내 카드나 다른 대체 콘텐츠를 대신 보여주면 안 된다.
- 위젯은 앱의 스토리 카드 편집 화면 진입점을 제공해야 한다.
- 위젯에서는 실제 편집 기능을 제공하지 않는다.
- 위젯은 상태 확인과 빠른 진입에 집중한다.

## 6. 알림 요구사항

다음 알림이 필요하다.

### 6.1 상대 스토리 카드 업로드 알림

- 상대가 오늘의 스토리 카드를 업로드하면 사용자에게 알림을 보낸다.

### 6.2 질문 생성 완료 알림

- 두 사람 모두 스토리 카드를 올려 질문이 생성되면 알림을 보낸다.

### 6.3 상대 답변 완료 알림

- 상대가 생성된 질문에 대한 답변을 완료하면 알림을 보낸다.

## 7. 상태 전이 요구사항

하루 단위 진행 상태는 최소한 아래 흐름을 가져야 한다.

1. 아무도 작성하지 않음
2. 한쪽만 스토리 카드 작성 완료
3. 양쪽 모두 스토리 카드 작성 완료
4. 질문 생성 중
5. 질문 생성 완료
6. 한쪽만 답변 완료
7. 양쪽 모두 답변 완료

- 상태 전이는 사용자가 현재 무엇을 해야 하는지 홈, 위젯, 알림에서 일관되게 반영해야 한다.

## 8. 데이터 및 동작 제약

- 하루 단위 공용 진행은 1개만 존재해야 한다.
- 각 사용자는 하루에 1개의 스토리 카드만 가질 수 있어야 한다.
- 두 사람 모두 완료되기 전에는 질문이 생성되면 안 된다.
- 생성된 질문은 해당 커플의 해당 날짜 하루 공용 루프에 1:1로 연결되어야 한다.
- 질문이 연결하는 루프는 반드시 같은 커플의 같은 날짜 루프여야 한다.
- 질문 생성 이후에는 스토리 카드가 잠겨야 한다.
- 질문 답변 데이터는 기존 텍스트 답변 구조를 유지해야 한다.
- 홈, 캘린더, 위젯은 모두 단순 사진이 아니라 스토리 카드 결과물을 기준으로 표시해야 한다.

## 9. 사용자 경험 목표

- 사용자는 질문을 기다리기보다 먼저 감정과 상황을 시각적으로 표현할 수 있어야 한다.
- 사용자는 상대가 올린 카드를 보고 자연스럽게 반응하고 싶어져야 한다.
- 질문은 대화의 시작점이 아니라, 시각적 교환 이후의 대화 확장 장치로 작동해야 한다.
- 앱은 커플 다이어리이면서 동시에 가벼운 일상 공유 도구처럼 느껴져야 한다.

## 10. 수용 기준

아래 조건이 만족되면 요구사항이 충족된 것으로 본다.

- 사용자는 세로형 4:5 폴라로이드 프레임에서 사진, 그림, 자유 텍스트 중 하나 이상으로 스토리 카드를 만들 수 있다.
- 사진은 1:1 영역에 표시되고, 선택 문구는 하단 고정 영역에 최대 50자와 2줄로 표시된다.
- 자유 텍스트와 드로잉은 폴라로이드 프레임 전체에 배치할 수 있다.
- 한쪽이 카드를 올리면 상대는 즉시 그 카드를 볼 수 있다.
- 두 사람 모두 카드를 올리면 질문이 생성된다.
- 질문 생성 후 카드 수정 및 삭제가 불가능하다.
- 질문 답변은 기존처럼 텍스트로 작성된다.
- 홈 화면은 질문 중심이 아니라 스토리 카드 상태 중심으로 동작한다.
- 홈 화면은 오늘 카드가 없는 날에도 최근 카드 폴백이 아니라 오늘의 빈 상태를 보여준다.
- 내가 먼저 오늘 카드를 올린 경우 홈은 내 카드와 상대 대기 상태를 함께 보여준다.
- 두 사람 모두 카드를 올린 직후 질문이 아직 완성되지 않은 짧은 구간에는 `질문 생성 중` 상태가 보인다.
- 캘린더는 날짜별 스토리 카드 기록을 보여준다.
- 캘린더 월간 셀을 누르면 현재 화면 안에서 해당 날짜 상세 패널이 갱신된다.
- 위젯은 오늘 카드가 없을 때 상대가 최근 올린 스토리 카드 미리보기를 포함한 앱 진입 기능을 제공한다.
- 위젯은 상대의 업로드 이력이 전혀 없을 때 대기 상태와 앱 진입 기능을 제공한다.
- 알림은 상대 카드 업로드, 질문 생성 완료, 상대 답변 완료를 지원한다.

## 11. 기능 분해

아래 기능 분해는 현재 요구사항을 실제 분석, 설계, 구현 단위로 나눈 것이다.

### 11.1 스토리 카드 도메인 기능

- 하루 단위 커플 공용 진행 엔티티를 정의해야 한다.
- 사용자별 하루 1개 카드 제약을 보장해야 한다.
- 카드 상태를 정의해야 한다.
  - 미작성
  - 한쪽 작성 완료
  - 양쪽 작성 완료
  - 질문 생성 완료
  - 답변 진행 중
  - 답변 완료
- 질문 생성 후 카드 잠금 규칙을 보장해야 한다.

### 11.2 스토리 카드 편집 기능

- 세로형 4:5 폴라로이드 프레임과 내부 1:1 사진 영역을 제공해야 한다.
- 앱 내부 카메라 우선 진입과 왼쪽 하단 갤러리 진입을 제공해야 한다.
- 사진 추가를 지원해야 한다.
- 자유 드로잉을 지원해야 한다.
- 캔버스 위치 지정, 이동, 확대·축소가 가능한 텍스트 레이어를 지원해야 한다.
- 스크롤 없는 전면 편집 화면과 오버레이 도구를 제공해야 한다.
- 변경 여부에 따른 폐기 확인 및 카메라 복귀 상태 전이를 제공해야 한다.
- 사진, 그림, 글 중 1개 이상 존재하는지 검증해야 한다.
- 저장을 지원해야 한다.
- 수정, 삭제, 재등록을 지원해야 한다.

### 11.3 스토리 카드 렌더링 기능

- 편집 결과를 하나의 카드 결과물로 표시해야 한다.
- 그림-only 카드 표시를 지원해야 한다.
- 글-only 카드 표시를 지원해야 한다.
- 사진-only 카드 표시를 지원해야 한다.
- 조합 카드 표시를 지원해야 한다.
- 홈, 캘린더, 위젯에서 공통으로 사용할 미리보기 결과물을 관리해야 한다.

### 11.4 하루 진행 상태 관리 기능

- 오늘 카드 존재 여부를 판단해야 한다.
- 상대 카드 업로드 여부를 판단해야 한다.
- 양쪽 카드 완료 여부를 판단해야 한다.
- 질문 생성 중 상태를 별도 관리해야 한다.
- 질문 생성 가능 시점을 판단해야 한다.
- 질문 생성 후 수정 차단 상태를 반영해야 한다.
- 답변 단계 진입 상태를 관리해야 한다.

### 11.5 질문 생성 연계 기능

- 기존 질문 placeholder 구조를 유지해야 한다.
- 질문 생성 시점을 두 사람 카드 완료 이후로 이동해야 한다.
- 질문이 먼저 존재하던 구조를 후행 생성 구조로 변경해야 한다.
- 생성된 질문을 기존 질문 화면 및 답변 화면과 연결해야 한다.

### 11.6 질문 답변 유지 기능

- 답변 형식은 기존처럼 텍스트만 유지해야 한다.
- 기존 답변 공개 규칙을 유지해야 한다.
- 두 사람 모두 답변해야 상호 공개되어야 한다.
- 기존 질문 상세 및 답변 저장 흐름의 재사용 범위를 명확히 해야 한다.

### 11.7 홈 화면 기능

- 오늘의 질문 중심 구조를 제거해야 한다.
- 방 구조와 캐릭터 중심 시각 배치를 유지해야 한다.
- 오늘의 스토리 카드 상태 중심 구조로 변경해야 한다.
- 스토리 카드를 홈 전면 대체 화면이 아니라 방 안의 중심 오브젝트로 배치해야 한다.
- 캐릭터 오브젝트와 스토리 카드 오브젝트를 서로 다른 sibling 구조로 유지해야 한다.
- 아무도 안 올린 날에는 오늘 카드의 빈 상태와 작성 진입을 표시해야 한다.
- 내가 먼저 올린 날에는 내 카드와 상대 대기 상태를 함께 표시해야 한다.
- 한쪽만 올린 날에는 상대 카드를 즉시 표시해야 한다.
- 양쪽 카드 업로드 직후에는 `질문 생성 중` 상태를 표시해야 한다.
- 양쪽 완료 후에는 질문 생성 상태를 표시해야 한다.
- 질문 생성 후 질문 화면 진입을 제공해야 한다.
- 홈에서 캐릭터 편집 진입을 계속 제공해야 한다.
- 홈 녹음 UI를 단일 메인 버튼 + 별도 보관함 진입 구조로 단순화해야 한다.
- 홈 녹음 UI에서는 카드형 테두리, 별도 재생 버튼, 상세 메타 정보 같은 보조 요소를 제거해야 한다.

### 11.8 홈 녹음 인터랙션 기능

- 홈의 녹음 인터랙션은 하나의 메인 버튼에서 재생과 녹음을 함께 처리해야 한다.
- 메인 녹음 버튼 짧은 터치는 현재 녹음 재생 / 일시정지를 담당해야 한다.
- 메인 녹음 버튼 길게 누르기는 새 녹음 시작을 담당해야 한다.
- 녹음 중 위로 드래그 취소 규칙은 유지해야 한다.
- 현재 녹음이 없는 상태에서 짧게 터치하면 사용자에게 짧은 피드백을 제공해야 한다.
- 현재 재생 중 길게 눌러 녹음을 시작하면 재생을 자동 중지한 뒤 녹음 단계로 전환해야 한다.
- 홈에서는 녹음 상세 메타 정보를 숨기고, 해당 정보는 보관함 화면에서 확인하도록 분리해야 한다.

### 11.9 캘린더 기록 기능

- 질문 중심 기록에서 스토리 카드 중심 기록으로 변경해야 한다.
- 날짜별 스토리 카드를 표시해야 한다.
- 해당 날짜의 생성 질문을 표시해야 한다.
- 해당 날짜의 답변을 표시해야 한다.
- 하루 기록 대표 단위를 질문에서 카드로 변경해야 한다.
- 월간 셀 탭 시 현재 화면 안에서 해당 날짜 상세 패널이 갱신되어야 한다.
- 오늘 날짜가 편집 가능 상태인 경우 캘린더 진입에서도 답변 수정 플로우로 이어질 수 있어야 한다.

### 11.10 위젯 기능

- 배경화면 위젯 미리보기를 제공해야 한다.
- 현재 스토리 카드 상태를 표시해야 한다.
- 홈 화면과 분리된 위젯 전용 폴백 미리보기 규칙을 가져야 한다.
- 오늘 카드가 없을 때는 상대가 최근 올린 스토리 카드를 미리보기로 표시할 수 있어야 한다.
- 상대가 한 번도 카드를 올린 적이 없을 때는 상대 카드가 없는 대기 상태를 표시해야 한다.
- 이 상태에서는 내 카드나 다른 콘텐츠로 대체하지 않아야 한다.
- 앱 편집 화면 진입을 제공해야 한다.
- 위젯 내 직접 편집은 제공하지 않아야 한다.

### 11.11 알림 기능

- 상대 스토리 카드 업로드 알림을 지원해야 한다.
- 질문 생성 완료 알림을 지원해야 한다.
- 상대 답변 완료 알림을 지원해야 한다.

### 11.12 권한 및 정책 기능

- 질문 생성 전 본인 카드만 수정 가능해야 한다.
- 질문 생성 전 본인 카드만 삭제 가능해야 한다.
- 질문 생성 후 수정 및 삭제가 불가능해야 한다.
- 상대 카드는 업로드 직후 열람 가능해야 한다.
- 하루 1인 1카드 제한을 유지해야 한다.

### 11.13 저장 구조 기능

- 스토리 카드 편집 원본을 어떻게 보관할지 정의해야 한다.
- 표시용 결과물 또는 미리보기 데이터를 어떻게 보관할지 정의해야 한다.
- 미리보기 결과물은 4:5 폴라로이드 비율로 저장하고 홈, 캘린더, 위젯에서 자르지 않고 표시해야 한다.
- 기존 9:16 미리보기 파일은 일괄 변환하지 않으며 새로 생성하거나 다시 저장하는 카드부터 폴라로이드 형식을 적용해야 한다.
- Storage 객체 경로 판별 함수는 `authenticated` 역할에만 실행 권한을 제공하고 `anon` 및 `public`에는 제공하지 않아야 한다.
- 홈, 캘린더, 위젯 공통 조회 구조를 정의해야 한다.
- 질문 및 답변과 스토리 카드 연결 구조를 정의해야 한다.

- 결제 연동
